Methods & Endpoints:
Core Runtime Components

The system boots up an in-process coordinator containing multiple autonomous background components:

  • Seat Hold Sweeper: Periodically sweeps active expired holds from the database (status transition: active → expired) and pushes updates to the SSE server layer.
  • Webhook Inbox Worker: Handles queued events sequentially inside transaction blocks using database locking (SKIP LOCKED) to avoid double-processing and CPU locks.
  • SSE Broadcast Node: Exposes the Server-Sent Events route (/api/seats/stream) fanning out global changes to clients in real-time.
Graceful Shutdown & Health Checks

Robust process shutdown hooks trap SIGTERM/SIGINT to guarantee database and queues are not left corrupted: 

// Order of operations:
1. Signal shutdown status -> Readiness fails (503)
2. Stop Seat Hold Sweeper loop
3. Allow Inbox Worker to drain current event loop
4. Close Prisma DB connections (disconnectPrisma)
5. Exit Node process
Ready checks (/api/health/ready) evaluate active database connections and background worker error logs, enabling container orchestrators to route traffic safely.

Db Index Invariants & Constraints:

Critical DB Constraints (Postgres Partial Unique Indexes)

To guarantee complete data exclusivity without relying on expensive in-memory lock pools, the application applies raw partial unique indexes directly to Postgres columns. If two processes attempt conflicting actions, the loser's transaction is aborted at the database engine boundary:

Invariant Goal Underlying SQL Condition Effect Behavior
One active hold per seat CREATE UNIQUE INDEX ON seat_holds(seat_id) WHERE status = 'active' Rejects any attempt to create or switch status to 'active' for a seat that already has an unexpired hold.
One active hold per user CREATE UNIQUE INDEX ON seat_holds(user_id) WHERE status = 'active' Restricts authenticated users to exactly one active hold. Forcing them to release the previous hold to secure a new seat.
One confirmed reservation per seat CREATE UNIQUE INDEX ON reservations(seat_id) WHERE status = 'confirmed' Permanently blocks double-booking. Prevents multiple reservations from succeeding for a single physical seat.
At most one active payment checkout per hold CREATE UNIQUE INDEX ON payments(seat_hold_id) WHERE status IN ('checkout_created', 'processing') Blocks users from creating multiple Stripe checkout URLs concurrently for the same seat hold.
Key Decisions & Alternatives Trade-offs
Decision Topic Selected Architecture Alternatives Considered "Why" / Trade-off Rationale
Framework Boundary Modular Monolith in Next.js Separated Microservices / Serverless Handlers Small domain. Avoids network latency hops and distributed locks. Keeps Prisma logic and type safety closely bound to frontend components.
Exclusivity Authority Database Constraints (Postgres) Redis SETNX lock, In-Memory Semaphore block Transactions combined with partial indices represent a secure, bulletproof source of truth. Does not suffer from lock leakage issues.
Fulfillment Trigger Stripe Webhook Event + Inbox Worker Success Redirect URL callback page Users can close browsers or abort redirects. Driving confirmation through Stripe's webhook guarantees delivery and ensures correct payment records.
Idempotency Strategy Natural / Business Keys in models Universal client-generated Idempotency Table Models already possess unique properties (stripeEventId, paymentId, seatHoldId). A dedicated table would duplicate these invariants.
Auth Token Rules Rotating Refresh Cookies + JWT Session Long-lived stateless JWT / LocalStorage Tokens Opaque, rotated refresh tokens stored in HttpOnly cookies restrict token theft. Presenting a used token revokes the entire user session family.
